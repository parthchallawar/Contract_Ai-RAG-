ContractAI — Complete Interview Guide

Everything in plain English: what it does, how the RAG pipeline works, how the two AI models connect, what each important function does, and why it was built this way. Read it once before an interview and you should be able to explain any part of it from memory.

01

The 30-second pitch (say this first)

"ContractAI is a web app where you upload a legal contract — PDF, Word, or text — and it gives you an AI analysis tailored to your role: Investor, Legal, PM, or Partner. You can also chat with the document. The part that makes it different: it doesn't just dump the whole contract into an AI model. It runs a real RAG pipeline built from scratch — local embeddings plus BM25 keyword search, combined with Reciprocal Rank Fusion. Every dollar figure the AI reports is checked against the real document text before it's counted, and every quote the chat cites is checked to actually exist in the contract — fake quotes get dropped. I also built an evaluation harness that proves the hybrid retrieval beats plain keyword search, with real measured numbers." Opening line — memorize this one

03

Big picture — what happens when you upload a contract

  1. Upload. The browser sends the file to POST /api/contracts/upload. The server saves it, creates a contract record with status analyzing, and replies right away. The heavy work runs in the background. The frontend polls GET /api/contracts/:id/analysis every few seconds — it gets HTTP 202 ("still working") until the status flips to completed.
    Why async? A big PDF plus three AI calls can take 30–60 seconds. You never want an HTTP request hanging that long.
  2. Text extraction. extractTextFromFile() picks the right reader — pdf-parse for PDF, mammoth for Word, plain read for TXT. Then normalizeNumericSpacing() fixes a common PDF problem: extraction breaks $1,000 into $1 000 with a stray space. The fix collapses spaces sitting between digits.
  3. Indexing. ensureContractIndex() builds the search index (full detail in the RAG section). It first hashes the text with sha256. If the hash matches last time — say, the user only switched roles on the same file — the index is reused instead of rebuilt, which skips re-embedding the whole document.
  4. Analysis. analyzeDocumentText() runs three AI calls and merges everything into one flat analysis object shared by every role:
    • Money analysis — selectMonetaryExposureWithLLM + verifyMonetaryItems
    • Legal insights — generateLegalInsightsWithRAG
    • PM insights — generatePMInsightsWithRAG
    • Plus extractRiskSignals() — plain regex flags (e.g. "does this contract mention termination for convenience?") that need no AI at all
  5. Save. The result lands in an in-memory Map and in SQLite (write-through). The frontend's next poll returns 200 with the full analysis, and the view renders.
Design point worth repeating: if any single AI step fails — bad JSON, a rate limit — it falls back to safe defaults and logs a note in analysisWarnings. The whole analysis never crashes because one part failed. This is graceful degradation, and it shows up everywhere in this project.
04

How RAG works here (the heart of the project)

RAG stands for Retrieval-Augmented Generation. In plain terms: instead of sending the whole document to the AI, you first find the few most relevant pieces and send only those. Cheaper, fits small context windows, and gives more focused answers.

4.1 — Chunking (chunkText in server.js)

The contract text is cut into pieces of roughly 1000–1500 characters — but not blindly:

  • First try to split on headings and paragraphs (splitIntoBlocks) — a contract clause should stay together.
  • If a block is too big, split on sentence boundaries (splitBySentences) — never mid-sentence.
  • Small blocks get packed together (packBlocksIntoChunks) so chunks aren't tiny.
  • Each chunk gets 150–200 characters of overlap with the previous one (applyOverlap), so nothing at a chunk border gets lost.
  • Every chunk remembers its start offset in the original text, needed later for citation highlighting.
Say this"I use clause-aware chunking — split on headings first, sentences as fallback, never mid-sentence, with overlap so nothing falls between the cracks."

4.2 — Two search indexes get built (buildContractIndex in retrieval.js)

Every chunk goes into two independent indexes.

A) BM25 — keyword search

buildBM25Index / scoreBM25. BM25 is the classic search-engine ranking formula (the one Elasticsearch uses), implemented here by hand. It scores a chunk higher when it contains the query words, with two adjustments:

  • Rare words count more (IDF) — "indemnification" showing up is a stronger signal than "agreement," which is everywhere.
  • Long chunks don't win unfairly — length normalization with b = 0.75 — and repeating a word ten times doesn't give ten times the score — saturation with k1 = 1.5.

It uses the ln(1 + …) form of IDF so very common words never score negative. The tokenizer (tokenize) is contract-aware — it keeps things like net-30, 12.3, and section numbers as single tokens instead of tearing them apart.

B) Semantic embeddings — meaning search

embedTexts turns every chunk into a vector of 384 numbers using a small model called MiniLM (Xenova/all-MiniLM-L6-v2), running locally inside the Node process via ONNX — no API call, no cost, no data leaves the machine. Vectors that mean similar things point in similar directions, so "terminate the agreement at its convenience" and "cancel the contract whenever they want" land close together even though they share almost no words. All vectors are L2-normalized (length = 1) at embed time — a nice trick, because cosine similarity between normalized vectors is just a dot product: one multiply-add loop, very fast.

4.3 — Query time: hybrid retrieval with RRF (hybridRetrieve)

When a question comes in, from chat or from the analysis prompts:

  1. The query is scored against BM25 → ranking #1.
  2. The query is embedded with the same MiniLM model and dot-producted against every chunk vector → ranking #2.
  3. The two rankings are merged with Reciprocal Rank Fusion:
final_score(chunk) = 1/(60 + rank_in_BM25) + 1/(60 + rank_in_semantic)

A chunk ranked #1 in BM25 gets 1/61, #2 gets 1/62, and so on — same for the semantic list — then the two are added.

Why RRF instead of adding raw scores? BM25 scores and cosine scores sit on totally different scales — BM25 might read 15.2, cosine is bounded 0–1. RRF only cares about rank position, so there's no messy score normalization to get right. The constant 60 is the standard value from the original RRF paper — it keeps rank #1 from steamrolling everything else. A chunk that shows up in neither ranking is dropped entirely — never returned as padding.

Why hybrid at all? Contracts contain two different kinds of things. Exact strings — section numbers, "$50,000," defined terms like "Net-30" — where semantic search is weak and BM25 nails it. And paraphrases — the user asks "can they cancel anytime?" but the contract says "terminate for convenience" — where BM25 shares zero words and embeddings catch it instantly. Hybrid means neither method has to carry the whole job alone.

4.4 — The fallback ladder

Retrieval never crashes the app. Three levels, in order:

  1. Hybrid — BM25 + embeddings + RRF. The normal case.
  2. BM25-only — used if the embedding model failed to load, e.g. a first-run model download failed.
  3. Plain keyword scoring (scoreChunkByKeywords) — used if the whole index is missing.
Say this"Every layer of the system degrades instead of throwing — retrieval, database, AI provider, all of it."
05

Two AI models, one code path

The problem: the project runs on a free OpenRouter model (nemotron-3-nano-30b-a3b:free), and free means roughly 50 requests a day. During demos or testing, that limit gets hit and every AI feature starts returning errors.

The fix: an ordered provider list with automatic fallback, defined in server.js as LLM_PROVIDERS:

const LLM_PROVIDERS = [
  { name: 'openrouter', apiKey: OPENROUTER_API_KEY, model: OPENROUTER_MODEL, url: 'https://openrouter.ai/api/v1/chat/completions' },
  { name: 'nim',        apiKey: NIM_API_KEY,        model: NIM_MODEL,        url: 'https://integrate.api.nvidia.com/v1/chat/completions' }
].filter((p) => p.apiKey && p.model);
  • Order matters — OpenRouter is always tried first. NVIDIA NIM only kicks in if OpenRouter fails.
  • The filter — if NIM's key or model aren't set in .env, it simply isn't in the array, and the app behaves exactly like the single-provider version. Fallback is fully optional.
  • Same shape, both providers — both speak the "OpenAI-compatible" API format, same request JSON, same streaming format, so there's one code path — just different URL, key, and model per provider.
  • When fallback triggers — a 429 (rate limit) or 5xx (server error) moves to the next provider immediately, no point retrying a provider whose daily quota is dead. Other errors, like malformed JSON output, still retry the same provider first, since those are usually random.
  • Streaming rule — for the chat stream, fallback only happens before the first token reaches the user. Once tokens are flowing, switching models mid-sentence would produce a mixed, confusing answer, so a mid-stream failure just shows an error instead.
  • JSON-mode tracking is per provider (jsonModeUnsupportedByProvider) — if OpenRouter's model rejects response_format: json, that says nothing about NIM's model, so each is remembered separately.
Proof it worksTested live — OpenRouter's real daily limit got hit, real 429s — and both the full analysis pipeline and the streaming chat fell back to NVIDIA NIM (meta/llama-3.1-8b-instruct) and produced correct output with verified citations.
06

Numeric grounding — "the AI never invents a dollar amount"

LLMs are bad at arithmetic and sometimes invent numbers outright. In a contract analysis tool, one invented dollar figure is a disaster. The fix is a three-step pipeline where the AI never does math and never gets the last word.

  1. extractMonetaryCandidates(text) — a deterministic JS regex scan finds every money-looking number in the text, then filters hard: bare years ("2024"), section references ("Section 5.2"), table-of-contents dot-leader lines ("Payment ......... 12"), numbered headings, document codes like "FMA-9" or "DGS-30-084." A number only survives if it has a currency symbol or a money word nearby — "fee," "penalty," "payment" — via hasMoneyKeywordNearby() / isNonMonetaryArtifact().
  2. selectMonetaryExposureWithLLM(candidates) — the AI's only job is to look at each candidate with its surrounding text and label it a risk (money that could be lost) or an obligation (money that must be paid). It classifies. It never adds numbers.
  3. verifyMonetaryItems(parsed, candidates) — every item the AI returned gets checked against the real candidate list: an exact raw match first ("$50,000" === "$50,000"), then amount-equality, which handles the AI reformatting "$50,000" as "50000 USD." Each real occurrence in the text can be consumed only once per category, so a fee listed twice by the AI gets deduplicated. If two items share an amount and their surrounding text overlaps more than 60% — Jaccard similarity on words, jaccardSimilarity() — the item gets a soft possibleDuplicate flag, shown to the user but not dropped, because two genuinely separate $10,000 fees can legally exist. Anything that can't be matched to real text is dropped and logged into analysisWarnings — it never touches the totals.
  4. JS sums the survivors. Every total is a plain JavaScript reduce(). The UI shows a grounding badge ("12/13 amounts verified") and a "show source" button per row that reveals the exact sentence the number came from.
Say this"LLM classifies, JS verifies and sums. Nothing invented by the model can reach a total."
07

Chat with verified citations

Memory across turns

The client sends the last 8 turns of conversation, but the server re-validates that history itself (sanitizeChatHistory) — length caps of 2000 characters per turn, role checks, dropping previous error messages. It never blindly trusts what the client sends.

The follow-up trick: the retrieval query combines the last user question with the current one. Why? "Is that normal?" on its own retrieves nothing useful — but paired with the previous question, "What's the late payment penalty?", retrieval finds the right chunks.

Answer format — plain text, not JSON

The model replies in a fixed layout:

<the answer text>
SOURCES:
- "exact quote from contract"
IMPLICATIONS:
- practical consequence

Why not JSON? Free and small models frequently break JSON when writing conversational text — unescaped quotes, cut-off strings. A plain-text protocol with headers is far more robust, and parseChatAnswer() splits it tolerantly — a missing section just yields an empty list, never a crash.

Citation verification — verifyQuote()

Every quote in SOURCES gets checked against the real contract text:

  1. Normalize both sides — lowercase, curly quotes to straight quotes, whitespace collapsed to single spaces. Models often change these cosmetically.
  2. Offset mapping — while normalizing, buildNormalizedOffsetMap() remembers where each normalized character came from in the original text, so a match found in normalized space can point back to the exact position in the original document. That position powers click-to-highlight.
  3. 8-word-prefix retry — models often start a quote correctly and then pad the ending with invented words. If the full quote fails to match, retry with just its first 8 words — if that matches, keep the real matched portion.
  4. Still fails? Dropped. An unverifiable quote is never shown, not even with a warning label — showing "maybe-fake" quotes would destroy trust in the real ones.

The frontend keeps a mirror copy of this same normalization logic (normalizeForQuoteMatch / findQuoteOffsetWithFallback in app.js), so clicking a citation chip opens the extracted-text panel and highlights the exact quoted sentence in <mark id="citation-highlight">.

Streaming (SSE)

POST /api/chat/stream sends tokens one at a time over Server-Sent Events, so the answer appears word by word, like ChatGPT. parseSseChunk() is a line buffer — the network can split a data: {...} line in half between packets, so partial lines get buffered until the newline arrives. The frontend writes each token straight into one DOM node by id (streaming-msg-content) instead of re-rendering the whole page per token, which would rebuild the full page hundreds of times. One full render happens at the end. If streaming produces zero tokens, the frontend silently falls back to the normal non-streaming POST /api/chat.

A real bug worth tellingInitially I used req.on('close') to detect the user closing the tab, in order to abort the AI call. But in Express, req's 'close' event fires as soon as the request body finishes reading — which killed every stream instantly. The fix was res.on('close'), guarded by a check on res.writableEnded. A good "hard bug I debugged" story if asked.
08

Persistence — SQLite (backend/db.js)

  • Uses better-sqlite3 — synchronous, WAL mode.
  • Write-through + hydrate-on-boot. All reads still come from in-memory Maps — fast, zero code change to routes — and every write additionally goes to SQLite. On server start, hydrateFromDb() loads everything back into the Maps.
  • Tables: contracts, analyses (the JSON blob), chunks — chunk text plus an embedding stored as raw bytes, a Float32Array's buffer written directly and read back as a zero-copy view — chat_messages, versions.
  • BM25 is not stored — it's rebuilt from chunk text at boot, cheaper than serializing its internal maps.
  • A contract found at boot still marked analyzing means the process died mid-analysis, so it gets flipped to error.
  • Graceful degradation again: if better-sqlite3 fails to load — native module, platform issues — every db function becomes a no-op, /api/health reports persistence: 'disabled', and the app runs memory-only. The app must never fail to start because of the database.
09

Evaluation harness — "I have numbers, not vibes"

backend/eval/ contains 3 synthetic fixture contracts, written for this purpose and PII-free, with deliberate paraphrase traps — questions worded to share no vocabulary with the answer text (the question says "walk away," the contract says "terminate for convenience"). A 30-item gold dataset (dataset.json) has 24 retrieval questions with known-correct source passages plus 6 numeric questions. run-eval.js compares three retrieval modes on the exact same production code paths and the same index, so only the scoring method differs, not the chunking:

Modehit@4hit@6MRR
Keyword (baseline)87.5%95.8%0.728
BM25 only87.5%91.7%0.747
Hybrid (RRF)95.8%95.8%0.852

hit@4 — was a correct chunk in the top 4 results? MRR — mean reciprocal rank, which rewards putting the right answer higher: 1st place scores 1.0, 2nd scores 0.5, 3rd scores 0.33, and so on. Hybrid wins on hit@4 and MRR, especially on the paraphrase traps that keyword and BM25 miss by design.

Honesty points kept in the repo on purpose: keyword actually beats BM25 at hit@6 on this specific dataset — a real result, kept, not hidden. And the dataset was written before any tuning; a 30-item set proves direction, not statistical significance.

Default mode is fully offline and deterministic, about 1 second. An optional --llm flag also measures numeric-grounding rate and citation-verification rate, which needs API access and isn't deterministic.

Say this"I committed a baseline JSON with real measured numbers, including the result that doesn't flatter my system. That's the difference between an eval and a marketing slide."

What makes this project unique — your talking points

  1. 1
    Real RAG, built by hand. BM25 written from the formula, local ONNX embeddings, RRF fusion. Not a LangChain wrapper — every line is explainable.
  2. 2
    Verification everywhere. Money figures are grounded against source text; chat quotes are verified or dropped. The AI never gets the final word on a fact.
  3. 3
    Measured, not claimed. A committed eval harness with a gold dataset and baseline numbers, including one result that doesn't favor the system — kept in, for honesty.
  4. 4
    Graceful degradation as a philosophy. Embedder fails → BM25-only. Index fails → keyword. SQLite fails → memory-only. OpenRouter rate-limited → NVIDIA NIM. One AI step fails → safe defaults plus a warning. The app never hard-crashes over a dependency.
  5. 5
    Two-provider AI integration. Automatic OpenRouter → NIM fallback with correct streaming semantics — never switch models mid-stream.
  6. 6
    Local embeddings. Zero cost, no rate limits on indexing, document content stays on the machine — a real privacy story for legal documents.
  7. 7
    No vector database, on purpose. A dot product over a few hundred pre-normalized 384-dim vectors is sub-millisecond. A vector DB would solve a scale problem this app doesn't have — knowing when not to add infrastructure is the point.
12

Likely interview questions & answers

Why RAG instead of just sending the whole contract to the model?

Three reasons. Cost — you pay per token, and a 50-page contract is huge. Context limits — small or free models have small windows. Quality — models get distracted by irrelevant text, the "lost in the middle" problem, so giving them only the 4–6 most relevant chunks sharpens the answer. Retrieval also gives exact source positions for verified, clickable citations — something you can't do from a single blob answer.

Why hybrid retrieval? Why not just embeddings?

Contracts are full of exact strings — "$50,000," "Section 12.3," "Net-30" — that semantic search under-ranks because it matches meaning, not characters. BM25 nails exact terms; embeddings catch paraphrases like "cancel anytime" versus "terminate for convenience." RRF combines their rank lists so neither has to be right alone. The eval proves it: hybrid hit@4 is 95.8% versus 87.5% for either method alone.

Why RRF instead of a weighted score combination?

BM25 and cosine scores sit on incompatible scales, which would need score normalization and a tuned weight. RRF only uses rank positions, so it needs no normalization and no tuning — and it's the standard, well-studied approach.

Why no vector database, like Pinecone?

Scale. Each contract produces tens to a few hundred chunks. A brute-force dot product over pre-normalized vectors is sub-millisecond. A vector DB solves problems — millions of vectors, ANN search, cross-document search — that this project doesn't have. It would earn its keep if the search spanned thousands of contracts at once.

How do you stop the AI from hallucinating numbers?

The AI never produces numbers on its own. JS regex extracts real candidates from the text; the AI only labels them risk or obligation; every labeled item must then match back to real source text — exact match, then amount-equality — or it's dropped and logged. All sums are plain JavaScript. Duplicate consumption also stops a restated fee from being counted twice.

How do citations work?

The model returns quotes in a SOURCES section. Each quote is normalized — case, curly quotes, whitespace — and searched inside the equally normalized contract text, with an offset map back to original positions. If the full quote fails, it retries the first 8 words, since models tend to pad quote endings with invented text. If nothing matches, the quote is dropped, never shown. The matched offset powers click-to-highlight in the UI.

What was the hardest bug?

The streaming abort. I used req.on('close') to abort the LLM call when the client disconnects — but in Express that event fires as soon as the request body finishes reading, so every stream got killed before the first token. Found it by logging inside the close handler, then switched to res.on('close') guarded by res.writableEnded.

How does the second AI provider work?

An ordered LLM_PROVIDERS array — OpenRouter first, NVIDIA NIM second, only present if its key and model are configured. Both are OpenAI-API-compatible, so it's one code path with per-provider URL, key, and model. A 429 or 5xx skips straight to the next provider; for streaming, fallback only happens before the first token so the user never gets an answer stitched from two models. Verified live against real OpenRouter 429s.

What would you improve with more time?

A paid model for reliable structured output; a bigger eval set, since 30 items shows direction, not significance; conversation summarization instead of a hard 8-turn truncation; a schema migration runner for SQLite; multi-currency handling, since mixed currencies currently sum face values; and auth/multi-user support for real deployment.

Why plain-text protocol for chat instead of JSON?

Free and small models constantly break JSON when writing conversational prose — unescaped quotes, truncation. A fixed plain-text layout, answer / SOURCES: / IMPLICATIONS:, parsed tolerantly, is far more robust. JSON-mode is still used for the analysis calls, where the output is short and structured.

What's the frontend architecture?

Deliberately zero-framework — one state object, and a render() that rebuilds the page HTML on every state change. One exception: streaming tokens write directly to a single DOM node, since re-rendering the whole page per token would be wasteful. All AI/user text passes through escapeHtml before touching innerHTML for XSS protection.

Honest limitations — say these before they ask

  • Free default model means 50 requests a day and flaky structured output — the NIM fallback mitigates the quota, a paid model would fix the quality.
  • MiniLM has a roughly 256-token input window — very long clauses lose some semantic (not lexical) context.
  • Mixed currencies are summed at face value, no FX conversion — documented, not silent.
  • A 30-item eval is directional evidence, not statistical proof.
  • Single-process, no auth, one SQLite file — a local/demo tool, not a deployed product.
